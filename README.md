## Question and Background
	The question we built our model to answer is: Can the characteristics of Minecraft mobs be used to predict whether a mob is hostile or peaceful. Dividing mobs into two groups, we chose to use classification to determine the outcome using a Kaggle dataset on Minecraft mobs. Mobs are any other moving figure in Minecraft that is not a player, for example, pigs, zombies, spiders, villagers, etc. Mob behavior is imperative in Minecraft gameplay as a mob can threaten players, or remain passive. By analyzing the relationship between players and mobs along with other details, we tested whether game design patterns can be learned from data through machine learning. 
	
	Using the mob data from Kaggle, we used features explaining the design of mob behavior. These included variables such as health, damage, spawning, reproduction, etc.. Additionally, we paired our research with the Data Science lifecycle which provided specific questions, exploratory analysis, machine learning models, and evaluation of these models. Connecting to the course, we used a Decision Tree and a Random Forest to show the final solution of our model. 

## Exploratory Data Analysis
The exploratory data analysis stage was used to understand the structure of the dataset and identify which variables might matter most for predicting mob behavior. We first examined the size of the dataset, data types, missing values, and basic summary statistics. This helped show which columns were numerical, which were categorical, and where cleaning or transformation would be needed before modeling. Rather than only producing graphs for the sake of visualization, the purpose of this stage was to build a story about which gameplay-related traits appear to separate Hostile mobs from Peaceful mobs.
	
The exploratory Data Analysis helps us understand the structure of the dataset and which variables matter the most for predicting mob behavior. With limited data because of the finite amount of mobs in the game, we took into account the size of the dataset, data types and missing values. We classified which categories were numerical, categorical, and which parts needed cleaning or transformation. During this stage, the main focus was to see which gameplay-related traits differentiated hostile vs peaceful mobs. 
	
The target variable, behaviorTypes allowed us to build our model and classification around it. We combined the Conditional class (a behavior category) with the Hostile class because mobs acting on conditional movements behave like threats depending on the situation. Another reason to do this is because our goal was to have a simple binary classification task. Additionally, we removed columns such as mob name and ID as they functioned as an identifier instead of a predictor, allowing the model to have a more clear prediction problem/answer. 

Feature engineering played an important role in the EDA step․ Missing values for maxDamage were filled with 0‚ because anytime a max damage variable has no value‚ it means the mob does not do direct attacks․ The variable reproductiveRequirement was simplified to a binary value indicating whether or not a mob can reproduce․ Finally‚ the debutDate variable was changed to debutYear so as to leverage the information about time in the model in numeric form․ In general the EDA indicated that damage‚ spawning and reproduction likely provided the most information for classification․ The EDA therefore helped justify the approach and helped answer the project question with respect to the raw data․

## Methods
We used a supervised machine learning approach to solve the project question‚ where the target variable was the behaviorTypes variable‚ and all other cleaned and engineered variables were used as predictors․ To conform with model building best practices‚ the data was randomly partitioned into a training‚ validation‚ and testing set in order to decouple the model building process from the final model performance in the data science life cycle․
After the project adopted the pipeline method required by the rubric‚ OneHotEncoder was applied to the categorical variables and preprocessing was combined with the classifier in a single pipeline․ This allows the pipeline to remain organized‚ reproducible‚ and consistent between models․ It also prevents data leakage from occurring since the same steps are taken for both the training and test data․

Models were built iteratively․ The first‚ Model 1‚ was a simple Decision Tree Classifier (DTC) because it is interpretable and can cope well with nonlinear relationships․ The model performed very well on the training set and poorly on the test set․ This may signal overfitting․ Model 2 was a GridSearchCV-tuned version of the decision tree․ Hyperparameters‚ such as the maximum depth of the trees‚ the split criterion‚ the minimum samples required to split and the minimum samples at leaf node‚ were altered in order to improve the generalization of the model․ The 3rd model was a Random Forest Classifier‚ which is an ensemble of many decision trees․ This model was selected due to having lower overfitting and better overall performance‚ and by making multiple models and comparing their results‚ this satisfied the rubric's requirement for iteration and explanation of how the best solution was arrived at․

## Evaluation of the Model
The results obtained by each of the models during the testing phase were then studied: for this classification problem‚ accuracy‚ precision‚ recall‚ F1-score‚ confusion matrices and ROC-based analysis were used as performance metrics to evaluate the models‚ when appropriate․ These values provide a better understanding than just accuracy‚ determining how well the model can differentiate between Hostile mobs and Peaceful mobs․

The test accuracy for the baseline decision tree was 81․25%․ The tuned decision tree also achieved 81․25%․ Both models performed reasonably well based on classification reports‚ but still made many classification errors and were not very generalizable․ The decision tree model seemed to predict Hostile mobs better than Peaceful mobs․ This is helpful‚ but it is still worth noting that prediction accuracy differs between the two classes‚ as this is not just about being able to make predictions‚ but being able to make predictions confidently and accurately for each of the two classes․

The Random Forest is the best performing method based on the test data‚ with a test accuracy of 93․75%‚ and good precision and recall values in both classes; only one of the test data was misclassified․ The Random Forest is the model that correlates the mob's properties and its behavior type the best․ While the cross-validation results were taken into consideration in developing the models‚ the held-out test set gets the most weight in the analysis since it is the set that best represents how well each of the models performs on unseen data․ The random forest model is the one that performs the best and gives the most reasonable answer to the project question․


## Conclusion
Utilizing the findings of methods and evaluation‚ this project concludes that Minecraft mob traits can be used to accurately predict whether a mob is Hostile or Peaceful․ Among the trained models‚ the random forest is the strongest model․ From these results‚ we can see that it is quite likely mob damage‚ mob spawning‚ and mob reproduction are all correlated with whether a mob is classified as Hostile or Peaceful‚ and so Minecraft mob behavior can be predicted through machine learning techniques․
However‚ these conclusions should always be kept within the context of the model's limitations․ Since the training dataset is small‚ results can be sensitive to the train-test split‚ and should be viewed as less generalizable than results with larger datasets․ Other project decisions simplified the data‚ such as placing Conditional mobs in the Hostile group‚ and considering reproduction to be simply yes or no․ However‚ this approach loses some subtlety‚ and the model is only as good as the underlying features․ Any tactical differences in the gameplay not reflected in the dataset will not be accounted for in the predictions․
The downside to these concerns is that‚ despite the issues discussed‚ this project has demonstrated the entirety of a data science pipeline: question definition‚ data exploration‚ feature engineering‚ model selection and tuning‚ validation on unseen test data‚ and careful interpretation of results․ Most of all‚ we can conclude that a mob's characteristics are sufficient for a model‚ even a random forest‚ to classify behavior reasonably well․
